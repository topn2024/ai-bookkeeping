## 说明书

### 发明名称

基于周期性消费识别的订阅管理和浪费检测方法

### 技术领域

[0001] 本发明涉及人工智能与消费管理技术领域,具体涉及一种基于周期性消费识别的订阅管理和浪费检测方法及系统,可应用于订阅服务管理、周期性消费追踪、浪费识别等场景。

### 背景技术

[0002] 随着订阅经济(Subscription Economy)的快速发展,用户平均订阅10+个服务(视频会员、音乐会员、云存储、健身房、软件订阅等)。据统计,用户平均有30-40%的订阅服务处于闲置或低使用状态,造成大量资金浪费。

[0003] **现有技术一(手动记录)**:用户手动记录订阅服务和续费日期。技术缺陷:(1)依赖用户记忆,容易遗漏;(2)无自动识别功能;(3)无使用状态评估;(4)无浪费检测;(5)无续费提醒。

[0004] **现有技术二(订阅管理应用)**:市场上存在专门的订阅管理应用(如Truebill、Bobby)。技术缺陷:(1)需要用户手动添加订阅;(2)无法自动识别周期性消费;(3)仅提供续费提醒,不评估使用状态;(4)无浪费量化;(5)不支持中国市场的支付方式。

[0005] **现有技术三(银行账单分析)**:部分银行提供账单分类功能。技术缺陷:(1)分类粗糙,无法识别订阅;(2)无周期性检测;(3)无使用状态评估;(4)无浪费提醒。

[0006] **现有技术四(时间序列分析)**:学术领域提出周期性检测算法(如自相关分析、傅里叶变换)。技术缺陷:(1)仅停留在理论层面;(2)未应用于订阅管理;(3)无使用状态评估;(4)无浪费检测机制。

[0007] 综上所述,现有技术存在以下共性技术问题:(1)无法自动识别周期性消费;(2)缺乏使用状态评估方法;(3)无浪费量化机制;(4)缺乏智能续费提醒;(5)不支持中国市场特点。

### 发明内容

#### 发明目的

[0008] 本发明的目的在于提供一种基于周期性消费识别的订阅管理和浪费检测方法及系统,解决现有技术中无法自动识别、缺乏使用评估、无浪费检测等技术问题。

#### 技术方案

[0009] 本发明提出一种**周期性识别 + 稳定性分析 + 使用评估 + 浪费检测 + 智能提醒**的五维订阅管理系统,包括:

##### 核心技术方案一:周期性消费自动识别

[0010] **订阅数据结构**:
```
SubscriptionRecord {
  id: UUID,
  user_id: String,
  merchant: String,              // 商户名称
  amount: Decimal(18,2),         // 金额
  cycle_type: Enum,              // 周期类型:周/双周/月/季/年
  cycle_days: Int,               // 周期天数
  first_date: Date,              // 首次消费日期
  last_date: Date,               // 最近消费日期
  transaction_count: Int,        // 交易次数
  stability_score: Float,        // 稳定性评分
  usage_status: Enum,            // 使用状态
  is_waste: Boolean,             // 是否浪费
  annual_cost: Decimal(18,2),    // 年度成本
  potential_saving: Decimal(18,2) // 潜在节省
}
```

[0011] **周期性识别算法**:
```
算法1:周期性消费识别
输入:用户交易历史 Transactions
输出:订阅列表 Subscriptions

识别流程:
1. 商户+金额分组:
   - 按商户名称分组
   - 同一商户按金额聚类(允许±5%误差)
   - 形成候选订阅组

2. 时间间隔分析:
   - 计算相邻交易的时间间隔
   - intervals = [t[i+1] - t[i] for i in range(n-1)]

3. 周期类型判定:
   - 周期:intervals ≈ 7天 (±2天)
   - 双周:intervals ≈ 14天 (±3天)
   - 月:intervals ≈ 30天 (±5天)
   - 季:intervals ≈ 90天 (±10天)
   - 年:intervals ≈ 365天 (±15天)

4. 周期性确认:
   - 至少3次交易
   - 间隔标准差 / 平均间隔 < 0.2 (变异系数)
   - 最近一次交易在3个周期内

5. 订阅创建:
   - if 满足周期性条件:
       create_subscription(merchant, amount, cycle_type)
```

##### 核心技术方案二:稳定性分析

[0012] **稳定性评分算法**:
```
算法2:订阅稳定性评分
输入:订阅记录 Subscription
输出:稳定性评分 StabilityScore ∈ [0, 100]

评分维度:
1. 金额稳定性(40分):
   - 计算金额变异系数:CV = std(amounts) / mean(amounts)
   - 评分:stability_amount = (1 - CV) × 40
   - CV < 0.05: 40分(高度稳定)
   - CV > 0.2: 0分(不稳定)

2. 时间稳定性(40分):
   - 计算间隔变异系数:CV_time = std(intervals) / mean(intervals)
   - 评分:stability_time = (1 - CV_time) × 40
   - CV_time < 0.1: 40分(高度稳定)
   - CV_time > 0.3: 0分(不稳定)

3. 持续性(20分):
   - 订阅持续月数:months = (last_date - first_date) / 30
   - 评分:continuity = min(months × 2, 20)
   - 持续越久,稳定性越高

综合评分:
StabilityScore = stability_amount + stability_time + continuity

稳定性等级:
- 高度稳定:≥80分
- 较稳定:60-79分
- 中等稳定:40-59分
- 不稳定:<40分
```

##### 核心技术方案三:使用状态估算

[0013] **使用状态评估算法**:
```
算法3:订阅使用状态评估
输入:订阅记录 Subscription, 用户行为数据 Behaviors
输出:使用状态 UsageStatus

评估方法:
1. 直接使用数据(如果可获取):
   - 视频会员:观看次数、观看时长
   - 音乐会员:播放次数、播放时长
   - 健身房:打卡次数
   - 云存储:上传/下载次数

2. 间接推断(无直接数据时):
   - 关联消费:订阅后是否有相关消费
   - 示例:健身房会员 + 运动装备消费 → 使用中
   - 示例:视频会员 + 无相关消费 → 可能未使用

3. 时间衰减模型:
   - 最近使用时间越久,使用概率越低
   - usage_probability = exp(-days_since_last_use / 30)

4. 使用状态分级:
   - 经常使用:每周使用≥2次
   - 偶尔使用:每月使用1-4次
   - 很少使用:每季度使用1-2次
   - 完全未使用:3个月无使用记录

5. 中国市场特殊处理:
   - 视频会员:考虑热播剧集周期
   - 外卖会员:考虑工作日vs周末
   - 共享单车:考虑季节因素(冬天使用少)
```

##### 核心技术方案四:浪费订阅识别

[0014] **浪费检测算法**:
```
算法4:浪费订阅检测
输入:订阅记录 Subscription, 使用状态 UsageStatus
输出:是否浪费 IsWaste, 潜在节省 PotentialSaving

检测规则:
1. 完全未使用(高优先级):
   - 条件:3个月无使用记录
   - 判定:浪费
   - 节省:年度成本 × 100%

2. 很少使用(中优先级):
   - 条件:每季度使用1-2次
   - 单次成本:年度成本 / 使用次数
   - 判定:if 单次成本 > 单次购买成本 × 2: 浪费
   - 节省:年度成本 - 单次购买成本 × 使用次数

3. 重复订阅(高优先级):
   - 条件:同类服务订阅多个
   - 示例:同时订阅爱奇艺+腾讯视频+优酷
   - 判定:浪费
   - 节省:保留1个,取消其他

4. 功能重叠(中优先级):
   - 条件:不同服务功能重叠
   - 示例:百度网盘+阿里云盘(都是云存储)
   - 判定:部分浪费
   - 节省:合并到一个服务

5. 价格不合理(低优先级):
   - 条件:价格高于市场平均价
   - 判定:if 价格 > 市场价 × 1.5: 浪费
   - 节省:价格 - 市场价

年度节省潜力:
PotentialSaving = Σ(浪费订阅的年度成本)
```
