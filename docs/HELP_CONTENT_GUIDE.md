# 帮助内容编写规范

本文档说明如何为应用添加和维护帮助内容。

## 目录结构

```
app/assets/help/
├── home.json              # 首页与快速记账
├── statistics.json        # 统计报表
├── accounts.json          # 账户管理
├── budget.json            # 零基预算
├── voice.json             # 语音交互
├── settings.json          # 设置中心
└── ...                    # 其他模块
```

## JSON 文件格式

每个模块的 JSON 文件包含以下结构：

```json
{
  "module": "模块ID",
  "moduleName": "模块名称",
  "moduleDescription": "模块描述（1-2句话）",
  "contents": [
    {
      "pageId": "/page-route",
      "title": "页面标题",
      "module": "模块ID",
      "description": "功能描述（1-2句话）",
      "useCases": [
        "使用场景1",
        "使用场景2"
      ],
      "steps": [
        {
          "title": "步骤标题",
          "description": "步骤详细说明"
        }
      ],
      "tips": [
        "使用技巧1",
        "使用技巧2"
      ],
      "relatedPages": ["/related-page-1", "/related-page-2"],
      "keywords": ["关键词1", "关键词2"]
    }
  ]
}
```

## 字段说明

### 必填字段

- **pageId**: 页面路由，用于唯一标识页面
- **title**: 页面标题，简洁明了
- **module**: 所属模块ID
- **description**: 功能描述，1-2句话说明页面作用

### 推荐字段

- **useCases**: 使用场景列表（2-4个），帮助用户理解何时使用
- **steps**: 操作步骤列表（3-8步），提供详细的操作指导

### 可选字段

- **tips**: 使用技巧和注意事项（0-5个）
- **relatedPages**: 相关页面链接（0-5个）
- **keywords**: 搜索关键词（用于优化搜索）

## 内容编写原则

### 1. 用户友好

- ✅ 使用通俗易懂的语言
- ✅ 避免技术术语和专业词汇
- ✅ 关注用户的实际需求和使用场景
- ❌ 不要涉及技术实现细节
- ❌ 不要使用代码或API术语

### 2. 内容完整

每个页面的帮助内容应包含：
- 功能是什么（description）
- 什么时候用（useCases）
- 怎么使用（steps）
- 注意什么（tips）

### 3. 结构清晰

- 使用分步骤的方式说明操作流程
- 每个步骤包含标题和详细说明
- 使用场景要具体，避免抽象描述

### 4. 保持一致

- 统一的术语使用（如"记账"而不是"添加交易"）
- 统一的格式和风格
- 统一的语气（友好、专业）

## 内容模板

### 基础页面模板

```json
{
  "pageId": "/example",
  "title": "示例页面",
  "module": "example",
  "description": "这是一个示例页面，用于演示帮助内容的编写格式",
  "useCases": [
    "场景1：具体的使用场景描述",
    "场景2：另一个使用场景"
  ],
  "steps": [
    {
      "title": "第一步",
      "description": "详细说明第一步要做什么"
    },
    {
      "title": "第二步",
      "description": "详细说明第二步要做什么"
    }
  ],
  "tips": [
    "提示1：实用的使用技巧",
    "提示2：需要注意的事项"
  ],
  "relatedPages": ["/related-page"],
  "keywords": ["关键词1", "关键词2"]
}
```

## 添加新的帮助内容

### 步骤1：确定模块

确定新页面属于哪个模块，如果是新模块，需要创建新的 JSON 文件。

### 步骤2：编写内容

按照上述模板和规范编写帮助内容。

### 步骤3：验证格式

使用 JSON 验证工具确保格式正确：

```bash
python3 -m json.tool your_file.json
```

### 步骤4：测试

在应用中测试帮助内容是否正确显示：
1. 重启应用
2. 进入帮助页面
3. 搜索或浏览到新添加的内容
4. 检查显示效果

## 更新现有内容

1. 找到对应的 JSON 文件
2. 定位到要更新的页面（通过 pageId）
3. 修改相应字段
4. 验证 JSON 格式
5. 测试更新效果

## 最佳实践

### 功能描述

✅ 好的描述：
- "查看和管理所有账户，一目了然地掌握各账户余额"
- "通过语音描述完成记账，AI自动识别金额、分类和备注信息"

❌ 不好的描述：
- "账户管理页面"（太简单）
- "本页面提供账户的CRUD操作"（技术术语）

### 使用场景

✅ 好的场景：
- "每天打开应用查看账户总览"
- "购物后有小票或发票"

❌ 不好的场景：
- "需要查看数据时"（太抽象）
- "调用账户查询API"（技术实现）

### 操作步骤

✅ 好的步骤：
```json
{
  "title": "选择账户类型",
  "description": "选择账户类型：现金、储蓄卡、信用卡、支付宝、微信等"
}
```

❌ 不好的步骤：
```json
{
  "title": "点击按钮",
  "description": "点击"（缺少具体说明）
}
```

## 常见问题

### Q: 如何决定内容的详细程度？

A: 核心功能应该提供详细的步骤说明，次要功能可以简化。以用户的实际需求为准。

### Q: 是否需要为每个页面都添加帮助内容？

A: 建议为所有用户可见的页面添加帮助内容，但可以分优先级逐步完善。

### Q: 如何保持内容的时效性？

A: 功能更新时应同步更新帮助内容，建议在代码审查时检查。
