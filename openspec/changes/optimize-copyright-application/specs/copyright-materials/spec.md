## 新增需求

### 需求：软件著作权申请材料管理
系统必须提供完整、规范的软件著作权申请材料，确保材料符合国家版权局的申请要求，提高申请成功率。

#### 场景：软件基本信息规范
- **当** 准备软件著作权申请材料时
- **那么** 软件名称必须规范、无冗余，版本号和日期必须准确反映当前状态
- **并且** 软件分类、开发方式、权利取得方式等信息必须准确填写

#### 场景：功能说明完整准确
- **当** 编写软件功能说明时
- **那么** 必须完整描述所有核心功能模块
- **并且** 功能描述必须与实际代码实现一致
- **并且** 必须突出软件的创新点和技术特色

#### 场景：技术特点突出创新
- **当** 描述软件技术特点时
- **那么** 必须突出AI技术应用（图片识别、语音识别）
- **并且** 必须强调创新算法（零基预算、钱龄计算）
- **并且** 必须准确描述技术架构和核心技术

#### 场景：源代码文档规范
- **当** 准备源代码文档时
- **那么** 必须提供前30页和后30页源代码，每页50行
- **并且** 源代码必须选择能充分展示软件特点的核心代码
- **并且** 代码必须包含清晰的文件路径和页码标注
- **并且** 代码格式必须统一规范

#### 场景：代码统计准确
- **当** 统计代码规模时
- **那么** 必须使用自动化工具（如cloc）进行统计
- **并且** 必须排除第三方库和自动生成代码
- **并且** 必须分别统计前端、后端、管理后台的代码量
- **并且** 统计数据必须与实际代码一致

#### 场景：界面截图完整
- **当** 准备软件界面截图时
- **那么** 必须提供6-8张关键功能界面截图
- **并且** 截图必须使用PNG格式，分辨率适当
- **并且** 截图必须展示真实的软件界面和数据
- **并且** 每张截图必须有清晰的说明文字

#### 场景：材料审核检查
- **当** 完成申请材料准备后
- **那么** 必须使用检查清单进行全面审核
- **并且** 必须验证所有材料的完整性和准确性
- **并且** 必须确保格式符合申请规范
- **并且** 必须进行多轮审核以发现和修复问题

### 需求：源代码选择指南
系统必须提供源代码选择指南，指导如何选择最能展示软件特点的代码。

#### 场景：前30页代码选择
- **当** 选择前30页源代码时
- **那么** 必须包含应用入口文件（main.dart）
- **并且** 必须包含AI识别服务代码（图片识别、语音识别）
- **并且** 必须包含核心业务逻辑代码（记账、预算管理）
- **并且** 代码必须具有连续性和完整性

#### 场景：后30页代码选择
- **当** 选择后30页源代码时
- **那么** 必须包含创新算法实现（零基预算、钱龄计算）
- **并且** 必须包含数据处理和同步机制
- **并且** 必须包含后端API接口实现
- **并且** 代码必须能体现技术深度和复杂度

### 需求：申请材料检查清单
系统必须提供详细的申请材料检查清单，确保所有必需材料都已准备完整。

#### 场景：基本信息检查
- **当** 检查软件基本信息时
- **那么** 必须确认软件名称、版本号、日期等信息准确
- **并且** 必须确认申请人信息完整
- **并且** 必须确认软件分类和开发方式正确

#### 场景：文档材料检查
- **当** 检查文档材料时
- **那么** 必须确认功能说明完整准确
- **并且** 必须确认技术特点描述清晰
- **并且** 必须确认运行环境描述准确
- **并且** 必须确认功能结构图完整

#### 场景：源代码检查
- **当** 检查源代码文档时
- **那么** 必须确认前后各30页，共60页
- **并且** 必须确认每页50行代码
- **并且** 必须确认代码格式规范统一
- **并且** 必须确认文件路径和页码标注清晰

#### 场景：截图材料检查
- **当** 检查界面截图时
- **那么** 必须确认截图数量充足（6-8张）
- **并且** 必须确认截图格式和分辨率符合要求
- **并且** 必须确认截图内容展示关键功能
- **并且** 必须确认每张截图都有说明文字

#### 场景：最终提交检查
- **当** 准备提交申请材料时
- **那么** 必须确认所有材料已准备完整
- **并且** 必须确认所有格式符合规范
- **并且** 必须确认所有数据准确无误
- **并且** 必须确认申请表已正确填写和签署
